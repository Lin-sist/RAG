# design.md 追加约束：决策记录（Decision Record）

> 用途：在每个 Type C change 的 design.md 中固定产出一个「决策记录」小节，
> 供用户在事前闸门审查、以及后续用面试官 skill 逆向答辩时使用。
> 边界：本小节只写进 design.md，不写进 AGENT_LOG.md（AGENT_LOG 保持纯执行证据）。

## 对 codex 的指令

在 design.md 末尾新增一节 `## 决策记录`，为本 change 中**每一个存在真实岔路口的技术决策**各写一条。
判定标准：只要"当时有两种以上合理做法、最终选了一种"，就必须记一条；纯粹唯一解、无取舍的实现细节不必记。
每条严格用以下三行结构，用大白话，禁止贴大段代码：

### 决策 N：<一句话描述这个决策点>
- **面临的选择**：列出 2-3 个当时真实存在的候选方案（含最终被否决的），每个用半句话说清它是什么。
- **选了哪个 + 为什么**：写明选定项，并给出决定性理由（性能 / 职责边界 / 一致性 / 风险 / 与既有基线兼容等）。
- **放弃的代价**：说明被否决方案各自的具体代价或风险，即"如果选它会发生什么糟糕的事"。

## 硬性要求
1. 决策必须是**本 change 真实发生**的取舍，凡不确定的写"待用户在事前闸门确认"，不得编造备选方案凑数。
2. 每条决策必须可被一个"为什么不……"的追问检验；写完后自检：如果面试官问"你为什么不选另一个方案"，本条是否已给出答案。
3. 不复述 proposal 的用户故事，不复述 spec delta 的契约；本节只承载"取舍理由"。
4. out_of_scope 的边界决策（如"为什么不做 key 轮换 / 不做跨 provider fallback"）也应各记一条，因为这类"我刻意不做什么"恰恰是面试高频追问点。

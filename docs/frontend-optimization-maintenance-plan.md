# RAG 项目前端优化与长期维护文档

## 1. 文档说明

- 文档目标：作为当前 RAG 项目前端优化、设计协作、后续重构和长期维护的统一依据。
- 使用对象：项目维护者、前端开发者、UI/UX 设计师、Google AI Studio、Figma AI、Copilot/Codex 等辅助工具。
- 适用范围：`rag-frontend` 前端项目，同时约束与 `rag-admin` / `rag-auth` / `rag-core` / `rag-common` / `rag-document` 的接口协作边界。
- 当前版本基线日期：2026-03-15。
- 当前结论：后端主链路已具备支撑前端界面升级的基础稳定性，适合启动前端视觉与交互重构；但流式 RAG 元数据、一致性补偿和部分工程规范仍需继续演进。

## 2. 维护规则

### 2.1 状态枚举

- `TODO`：已确认问题或优化项，尚未开始
- `DOING`：正在处理
- `DONE`：已完成并验证
- `BLOCKED`：被后端能力、设计决策或外部资源阻塞
- `DEFERRED`：确认有价值，但当前阶段暂缓

### 2.2 更新要求

- 每次开始一个前端优化项前，先更新对应任务状态。
- 每次完成一个优化项后，补充“实施说明 / 验收结果 / 风险”。
- 若页面结构、交互模型、接口依赖、状态管理策略发生变化，必须同步更新本文件。
- 若设计改动依赖后端新增字段、接口语义变更或 SSE 协议升级，必须在本文件中显式记录依赖关系。
- 不允许只改 UI 不更新文档。

### 2.3 推荐维护字段

| 字段 | 说明 |
|------|------|
| 编号 | 问题或任务唯一标识，如 `FE-CHAT-01` |
| 优先级 | `P0 / P1 / P2 / P3` |
| 状态 | `TODO / DOING / DONE / BLOCKED / DEFERRED` |
| 模块 | 页面、组件、状态层或接口层 |
| 当前状态 | 当前实现的问题或局限 |
| 目标状态 | 改造后的预期表现 |
| 依赖 | 是否依赖后端、设计稿或新接口 |
| 验收标准 | 如何判断改造完成 |
| 备注 | 风险、兼容策略、后续演进 |

## 3. 当前项目真实状态

### 3.1 当前技术栈

- 前端：Vue 3.5 + TypeScript + Vite 6 + Pinia + Vue Router + Axios + Element Plus
- 聊天渲染：`markdown-it` + `highlight.js`
- 后端：Spring Boot 多模块，接口由 `rag-admin` 提供
- 认证：JWT Bearer，`accessToken + refreshToken`
- 异步任务：文档上传后返回 `taskId`，前端轮询任务状态
- 流式问答：前端通过 `fetch + ReadableStream` 消费 `SseEmitter` 输出

### 3.2 当前后端状态结论

基于 2026-03-15 的代码审查与本地测试结果：

- 后端 Maven 全量测试已通过，主链路处于“可回归、可迭代”的状态。
- 已具备稳定支撑前端重构的能力：
  - 登录、登出、刷新
  - 知识库 CRUD 与权限校验
  - 文档上传、异步索引、任务查询与取消
  - 同步问答
  - 流式问答
  - 历史记录与反馈
- 后端已经完成的重要工程化改造：
  - 当前用户身份提取统一化
  - 资源级权限校验
  - JWT 生命周期与 refresh 会话校验
  - 文档索引从 Controller 编排迁移到应用服务层
  - 异步任务 owner 授权校验
  - RAG 缓存键收敛
  - 限流、幂等、trace、请求观测落地

### 3.3 当前后端剩余限制

以下限制会直接影响前端设计方案，不应忽略：

- 流式问答当前只输出文本 chunk，不返回完整 `citations / contexts / metadata`。
- 流式问答历史沉淀时，当前保存的是空 citations，前端不能假设“流式回答结束后一定有完整引用来源”。
- 文档删除与索引处理在“向量库成功、数据库失败”或“数据库成功、向量库失败”的跨系统一致性补偿上仍不完整。
- 当前仓库存在未跟踪的 `application-dev.yml`，其中仍有开发态明文密码和 JWT secret；这不是线上阻塞项，但属于配置治理风险。

### 3.4 当前前端状态结论

前端不是空白项目，已经完成一轮基础建设和一轮局部设计升级，但整体处于“功能可用、视觉不统一、架构部分老旧”的混合状态。

已完成的部分：

- 路由、登录态、API 分层、类型定义、基础 store 已存在
- 核心页面已存在：
  - 登录页
  - 知识库列表页
  - 知识库详情页
  - 聊天页
  - 历史页
- 聊天页面已开始按 V2 设计方向升级：
  - 设计变量 `variables.css`
  - `ChatView.vue`
  - `ChatMessage.vue`
  - `CitationList.vue`
- Markdown、代码高亮、引用来源基础展示已接入

仍存在的明显问题：

- 设计系统落地不完整，项目中同时存在“V2 变量驱动样式”和“旧版硬编码管理台样式”
- 页面风格不统一，聊天页明显领先于知识库页和历史页
- 业务状态与交互反馈分散在页面内部，部分页面仍使用较重的视图级轮询/控制逻辑
- 前端对后端 RAG 元数据的使用仍偏浅，RAG 产品特征没有形成统一表达
- 移动端适配不完整，部分页面仍明显偏桌面后台布局

## 4. 当前前后端协作边界

### 4.1 当前前端可以稳定依赖的后端能力

- `POST /auth/login`
- `POST /auth/logout`
- `POST /auth/refresh`
- `GET /api/knowledge-bases`
- `POST /api/knowledge-bases`
- `GET /api/knowledge-bases/{id}`
- `PUT /api/knowledge-bases/{id}`
- `DELETE /api/knowledge-bases/{id}`
- `GET /api/knowledge-bases/{id}/statistics`
- `GET /api/knowledge-bases/{id}/documents`
- `POST /api/knowledge-bases/{id}/documents`
- `DELETE /api/knowledge-bases/{kbId}/documents/{docId}`
- `GET /api/tasks/{taskId}`
- `GET /api/tasks/{taskId}/result`
- `POST /api/tasks/{taskId}/cancel`
- `POST /api/qa/ask`
- `POST /api/qa/ask/stream`
- `GET /api/history`
- `GET /api/history/{id}`
- `DELETE /api/history/{id}`
- `POST /api/history/{id}/feedback`
- `GET /api/history/{id}/feedback`

### 4.2 当前前端不应假设存在的能力

- 流式问答完成后一定返回结构化 citations
- 流式问答中存在单独的 metadata 事件
- 问答接口提供稳定的“置信度”字段
- 文档任务接口提供完整的阶段级时间线或诊断详情
- 聊天会话具备真正的多轮会话 ID / 对话线程模型
- 知识库、文档、任务具备统一的前端可消费错误码字典

### 4.3 这对设计的直接影响

- 聊天页可以做强体验，但“流式引用来源卡片”“流式可信度条”“逐块来源联动”不能设计为强依赖后端实时数据。
- 引用来源展示应优先建立“同步问答体验完整、流式问答体验降级”的设计策略。
- 任务中心应以“状态轮询 + 成功/失败反馈 + 可取消”为主，不应假设后端已经提供丰富日志详情。
- 知识库、文档、历史三个域的 UI 重构可以立即开始，因为后端接口语义已经基本稳定。

## 7. 前端优化总目标

## 7.1 产品目标

- 将当前“功能可用的后台界面”升级为“企业级 AI 知识产品界面”
- 突出 RAG 产品特征：
  - 知识来源
  - 检索可信度
  - 文档处理状态
  - 知识库边界
- 建立从“知识库 -> 文档 -> 提问 -> 历史 -> 反馈”的完整体验闭环

## 7.2 工程目标

- 建立统一设计系统，减少页面硬编码样式
- 建立清晰的页面状态模型和交互反馈规则
- 页面尽量薄，复杂状态逻辑逐步下沉到 store / composable / domain 组件
- 使前端可以长期在不破坏主链路的情况下渐进式升级

## 7.3 协作目标

- 让外部设计工具可基于本文件直接理解当前真实项目状态
- 让设计方案明确区分：
  - 当前后端已支持的内容
  - 需要前端降级处理的内容
  - 需要后端补齐后再深挖的内容

## 7.4 非目标

- 本阶段不重写前端技术栈
- 本阶段不替换 Element Plus
- 本阶段不一次性重构所有页面
- 本阶段不假设后端新增会话系统、置信度字段或流式结构化事件

## 8. 设计与实现原则

### 8.1 统一设计系统原则

- 所有颜色、圆角、阴影、间距统一走 CSS 变量
- 不新增新的 UI 库
- 以现有 `variables.css` 为基线，不平行再造第二套视觉 token
- 逐步消灭页面级硬编码颜色与边距

### 8.2 渐进式重构原则

- 先统一设计系统，再统一布局，再逐页重构
- 先改“高频核心路径”，后改“低频管理界面”
- 每次只重构一个主页面或一个主交互链路

### 8.3 后端现实约束原则

- 任何依赖结构化流式数据的 UI，都必须标记为“后端增强后再完善”
- 任何需要稳定来源、可信度或诊断字段的设计，都应提供当前可降级版本
- 前端文档不夸大后端能力，不写“理想态冒充现状”

### 8.4 可维护性原则

- 样式优先抽到变量和复用类，不在页面中散落
- 复杂控制逻辑优先抽 composable
- 页面应清晰区分数据层、状态层、展示层

## 9. 推荐前端实施顺序

| 顺序 | 阶段 | 原因 |
|------|------|------|
| 1 | 设计系统收敛 | 不先收口 token 和公共样式，后续页面会继续分裂 |
| 2 | 全局布局与导航升级 | 这是所有页面统一产品感的入口 |
| 3 | 聊天页完善 | 核心体验页，最能体现 RAG 产品价值 |
| 4 | 知识库与文档页重构 | 补齐资产管理与异步处理体验 |
| 5 | 历史页与反馈闭环 | 形成问答审计链路的前端闭环 |
| 6 | 响应式、可访问性、观测 | 提升长期质量和上线稳定性 |

## 10. 分阶段优化路线图

## 10.1 阶段一：设计系统与全局框架统一

### 目标

- 收敛样式 token
- 统一品牌视觉
- 统一页面标题、空态、错态、卡片、按钮、抽屉、弹窗规范

### 建议改造

- 以 `variables.css` 为唯一设计 token 基线
- 清理知识库页、历史页中的硬编码颜色和固定像素样式
- 重构 `AppHeader.vue`、`AppSidebar.vue`、`DefaultLayout.vue`
- 统一页面标题区模板：
  - 标题
  - 简短说明
  - 主操作按钮
  - 次操作筛选

### 验收标准

- 新旧页面不再同时存在两套明显割裂的视觉语言
- 硬编码颜色数量显著下降
- 全局导航具备明确品牌感和当前页面定位感

## 10.2 阶段二：聊天页完善与 RAG 表达增强

### 目标

- 让聊天页成为全项目设计基准
- 强化 RAG 产品特征
- 明确流式和非流式两种能力边界

### 建议改造

- 保持当前聊天页的 V2 方向继续推进
- 强化输入区工具条：
  - TopK
  - 是否流式
  - 当前知识库标识
- 优化消息结构：
  - AI 消息操作区
  - Markdown 阅读性
  - Skeleton / 打字指示 / 错误态
- 同步问答场景下强化：
  - citations
  - contexts
  - metadata
- 流式问答场景下做降级设计：
  - 先展示文本
  - 结束后若无结构化来源，不强行渲染“伪引用”

### 验收标准

- 聊天页在无后端新增字段的前提下，仍具备完整、可信的交互闭环
- 同步问答和流式问答的视觉差异可解释
- 用户能够理解“当前回答是否有来源、来源是否完整”

## 10.3 阶段三：知识库与文档处理体验升级

### 目标

- 从“后台 CRUD”升级为“知识资产管理界面”
- 强化文档异步处理可见性

### 建议改造

- 重构 `KBListView.vue` 与 `KBCard.vue`
- 强化知识库卡片信息层级：
  - 名称
  - 描述
  - 文档数
  - 查询数
  - 公开/私有
  - 最后更新时间
- 重构 `KBDetailView.vue`
- 将文档处理状态显式区分：
  - 上传中
  - 排队中
  - 处理中
  - 已完成
  - 失败
- 评估是否加入任务抽屉或任务侧栏，避免进度只附着在详情页中部

### 验收标准

- 用户可以清楚看到知识库价值和当前处理状态
- 多文档上传时状态反馈明确
- 文档处理不是“黑盒”

## 10.4 阶段四：历史页与反馈闭环重构

### 目标

- 让历史页成为“问答审计与复盘页面”，而不是普通列表页

### 建议改造

- 重构 `HistoryView.vue`、`HistoryItem.vue`、`FeedbackDialog.vue`
- 统一和聊天页的阅读体验
- 强化历史详情中的：
  - 问题
  - 回答
  - 引用来源
  - 延迟
  - Trace ID
  - 反馈状态
- 若当前记录没有 citations，前端要明确表达“该记录未保存来源”而不是留白造成歧义

### 验收标准

- 历史页与聊天页形成统一阅读语言
- 用户可理解一条历史记录的完整上下文
- 反馈操作路径更短、更清晰

## 10.5 阶段五：响应式、可访问性与前端观测

### 目标

- 提升长期可用性和质量

### 建议改造
